我的 30 天 AI 编程学习路线

从 FastAPI 全栈到本地 LLM 部署，记录 30 天 AI 应用开发的完整学习路径
April 15, 2026 · 2 min · 225 words

目标：30 天掌握 AI 应用开发的核心技能，从能写 Web 后端到能跑本地大模型。

为什么开始这个计划#

学完基础 Python 后，我发现自己卡在"知道语法但做不出东西"的阶段。AI 浪潮来了，我想知道：

  • 怎么把大模型 API 接到自己的项目里？
  • 那些很酷的 Graph RAG、知识图谱到底怎么落地？
  • 本地跑 LLM 需要多强的机器？

于是制定了这份每天一个小项目的计划。

学习路线图#

Day 01 ──→ Day 02 ──→ Day 03 ──→ Day 04 ──→ Day 05 ──→ ...
全栈基础   AI 集成    知识图谱    模型评测    测试框架    持续学习

Day 01：全栈留言板（FastAPI + HTML/JS）#

目标：理解前后端分离架构。

用 FastAPI 写了一个极简留言板后端，前端用原生 HTML/JS 调用 REST API。核心收获：

  • Pydantic 数据模型自动校验
  • CORS 配置让前端能跨域访问
  • /docs 自动生成 Swagger 文档（FastAPI 真香）

Day 02：AI 语音助手（百度 ASR + DeepSeek + TTS）#

目标：打通"语音输入 → AI 思考 → 语音输出"的完整链路。

技术栈：

  • 百度语音识别（ASR）：把用户录音转成文字
  • DeepSeek API：生成 AI 回复
  • 百度语音合成（TTS）：把回复转成语音播报

踩坑记录：

  • 浏览器录制的 WebM 音频需要用 ffmpeg 转成 WAV（16kHz、单声道）才能被百度 ASR 识别
  • 百度 Token 有有效期，需要定时刷新

Day 03：Graph RAG 知识图谱（Neo4j + Kimi API）#

目标：理解结构化知识如何增强 LLM 问答。

用 Neo4j 构建了企业供应链图谱（宁德时代、比亚迪、特斯拉等节点），实现了 Graph RAG 问答系统：

用户问题 → 实体识别 → Neo4j 关系查询 → 上下文组装 → Kimi 生成回答

相比传统向量 RAG，Graph RAG 的优势：

  • 可解释性：能展示答案来源的图谱关系
  • 精确性：结构化查询避免语义漂移
  • 关系推理：支持多跳推理（A 的供应商的供应商）

Day 04：模型竞技场（llama.cpp 本地推理）#

目标：在本地 CPU 运行开源 LLM，理解模型量化与性能权衡。

测试了 Qwen0.5B、Qwen1.5B、Phi-2、TinyLlama 等模型。一个反直觉的发现：

7B 模型只比 0.5B 聪明约 10%，但体积大了 15 倍、加载慢了 20 倍。

对于边缘设备和简单任务，0.5B 模型的性价比远超预期。

Day 05：AI 综合能力测试框架#

目标：用工程化思维评估模型能力边界。

设计了一套包含 20 道题目的评测系统，覆盖逻辑、语言、数学、常识、创造力、伦理、元认知 7 个维度。框架采用插件化架构：

  • models/：统一接口封装 HuggingFace、OpenAI、Kimi、DeepSeek
  • questions/：可扩展的题目库
  • evaluator/：规则评分 + LLM 评判双机制
  • reporters/：控制台 / JSON / Markdown 多格式报告

这个项目的代码结构是我目前最满意的，真正理解了"可扩展架构"的含义。

给后来者的建议#

  1. 先跑通，再优化。Day 01 的留言板丑得掉渣，但它让我理解了前后端分离的核心概念。
  2. 环境隔离。每个项目独立虚拟环境 + .env 管理密钥，避免依赖冲突。
  3. 动手比等待重要。等 15GB 模型下载的焦虑，远大于写代码分析的快乐。
  4. 记录即收获。写博客的过程本身就是一次深度复习。
